ChatML Dialogue Template & Inference Prefix

Hybrid Mamba-TuckerMoE · §7.2 Heterogeneous Data Normalisation

§ 1   Unified Dialogue Format

All heterogeneous data sources are strictly mapped to the same ChatML dialogue syntax. This ensures that every token, regardless of its origin dataset, follows an identical structural grammar — eliminating format drift, role collapse, and multilingual garbage decoding observed in mixed-template experiments.

Single-turn Template

For single-turn question-answer pairs the normalised template is:

User
<|im_start|>user … user content …<|im_end|>
Assistant
<|im_start|>assistant … assistant answer …<|im_end|>

Multi-turn Template

Multi-turn dialogues concatenate multiple user–assistant pairs. The model sees the complete history as context but receives gradients only on assistant answer tokens.

User (Turn 1)
<|im_start|>user … first question …<|im_end|>
Assistant (Turn 1)
<|im_start|>assistant … first answer …<|im_end|>
User (Turn 2)
<|im_start|>user … follow-up question …<|im_end|>
Assistant (Turn 2)
<|im_start|>assistant … second answer …<|im_end|>

Figure T-1: ChatML-only dialogue template. <|im_start|> (id 32000) opens a role block; <|im_end|> (id 32001) closes it and — when following an assistant answer — receives a supervised label so the model learns when to stop generating.

§ 2   Inference Prefix — Open-ended Completion

At inference time we feed only the open assistant prefix and let the model continue generating. The terminal <|im_end|> token is withheld; the model must predict it to stop:

User (context)
<|im_start|>user … user content …<|im_end|>
Model continues from here ▼
<|im_start|>assistant

The dashed purple box marks the open-ended assistant prefix: the model has received the role boundary tokens but no answer content yet, so it must generate the full reply and end with <|im_end|> (id 32001) to signal completion.

$$ \underbrace{\texttt{<|im\_start|>}\,\texttt{assistant}\,\texttt{\textbackslash n}}_{\text{prefix (given, not supervised)}} \;\xrightarrow{\text{model generates}}\; \underbrace{w_1\;w_2\;\cdots\;w_T\;\texttt{<|im\_end|>}}_{\text{auto-regressive output}} $$

§ 4   Token Annotation Summary

Token / Region Input ID Label Role in SFT
<|im_start|> 32000 −100 Opens role block; always masked — model sees it as pure context
user / assistant (role tokens) BPE sub-tokens −100 Role identifier; part of the structural grammar, not the answer
\n (after role) 10 −100 Delimiter newline; masked along with the header
User question tokens BPE −100 Context only; gradient flows back through attention / SSM state but no direct loss
Assistant answer tokens BPE token_id Primary learning signal — all answer tokens supervised
<|im_end|> (after assistant) 32001 32001 Stop signal — supervised so the model learns when to stop
[PAD] 32006 −100 Batch-alignment padding; always masked

Table T-1: Token-level label assignment during ChatML-only SFT. The invariant that <|im_end|> is always supervised prevents the model from generating indefinitely after an assistant turn.
